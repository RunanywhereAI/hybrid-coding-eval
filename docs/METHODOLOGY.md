# Methodology

*Companion to `REPORT.md`. This document is the scoring, sourcing, and
limitations dossier — it exists so a reviewer (academic or
social-media-skeptic) can decide whether our numbers are credible before
reading the headline charts.*

*If you are here to run the experiment yourself, read `REPRODUCING.md`.
If you want the results, read `REPORT.md`. This file is about **how we
decided what to measure** and **what we are not claiming**.*

---

## 1. Overview

We built `hybrid-coding-eval` to answer one question: **on a real
developer's laptop, does routing some fraction of a coding task to a
local model save money without destroying quality?** To make that
question legible, we fix a hardware tier (Apple M4 Max, 64 GB, macOS
15), a single cloud model (`gpt-5.5`), a single local model
(`qwen3-coder:30b` via Ollama, quantized to MXFP8), three routing
strategies, and 30 tasks drawn from four benchmarks. We then run every
(task, route) pair exactly once, record tokens per backend, score
quality, and apply pricing downstream.

This document **does not** answer: whether hybrid routing is a good
idea in general; whether results transfer to other hardware, other
models, or other languages; whether the gap between frontier and
open-weights models has closed; or whether any specific cloud provider
is worth its sticker price. Our N is small (30 tasks, single sample
per cell), our hardware is one machine, and our routing heuristic is
deliberately simple. The numbers are a **well-documented data point**,
not a verdict.

## 2. Experimental design

| axis | levels | note |
|---|---|---|
| Categories | A (tiny), B (real SWE), C (arch / reasoning) | 4 benchmarks mapped to 3 categories — see §3 |
| Tasks per category | 10 / 10 / 10 | 30 total; 5 from BigCodeBench-Hard + 5 custom-arch in C |
| Routes | R1 cloud-only, R2 local-only, R3 hybrid-architect | see §4 |
| Samples per (task, route) | 1 | variance is NOT estimated |
| Hardware tier | 1: M4 Max 64 GB, macOS 15 | acknowledged limitation — see §7 |
| Primary metric | tokens (prompt, completion, cached) per backend | cost is derived, not primary |

Token counts are the primary recorded observable for three reasons:

1. they're deterministic (the proxy records exactly what was sent and
   received — see `lib/metrics.py::ResultRow`);
2. they're stable across pricing updates (we can re-cost the same
   runs a year from now as rates change);
3. they're the unit that actually differs between routes — quality
   is the *outcome*, tokens are the *behaviour*.

Pricing is applied downstream in `lib/pricing.py::compute_cost`, from a
pinned pricing table (`lib/pricing_tables.json`, dated 2026-04-27).
Multi-scenario sweeps over that table drive the sensitivity analysis
in the report.

## 3. Benchmarks — provenance and contamination risk

### 3.1 Category A — HumanEval+ (10 tasks)

| field | value |
|---|---|
| upstream | https://github.com/evalplus/evalplus |
| paper | Liu et al., *Is Your Code Generated by ChatGPT Really Correct?* — NeurIPS 2023, [arXiv:2305.01210](https://arxiv.org/abs/2305.01210) |
| license | HumanEval MIT (OpenAI); EvalPlus test extensions Apache-2.0 |
| adapter | `benchmark/humaneval_plus/adapter.py` |
| sample | `seed=42`, 10 tasks drawn from the 164-problem set |
| scoring | functional (§5.1) |
| **contamination risk** | **HIGH** — HumanEval was released 2021, EvalPlus 2023. Every model we benchmark (GPT-5.5, Qwen3-Coder) was trained on web crawls that include the problems and their solutions. |

**Why we keep it:** a memorization-friendly floor is useful. If hybrid
routing loses to cloud-single on tasks where the model has the answer
memorized, it shows the transport and router overhead is legible at
the low end. Category A is a **sanity check**, not a showcase.

### 3.2 Category B — SWE-bench Verified, easy tier (10 tasks)

| field | value |
|---|---|
| upstream | https://www.swebench.com/ (Verified split) |
| paper | Jimenez et al., *SWE-bench: Can Language Models Resolve Real-World GitHub Issues?* — ICLR 2024, [arXiv:2310.06770](https://arxiv.org/abs/2310.06770); Verified subset from OpenAI (2024) |
| license | MIT |
| adapter | `benchmark/swebench_verified/adapter.py` |
| sample | 10 tasks labeled `difficulty="easy"` (human annotation: "<15 min fix"), seeded selection |
| scoring | SWE-bench harness (§5.2) |
| **contamination risk** | **MEDIUM** — the tasks are real GitHub PRs from Django, Sphinx, Astropy, xarray, etc. Some appear in older training sets; the Verified subset re-curation and timestamps reduce but don't eliminate the risk. |

**Why we chose easy-tier and only 10:** the scoring harness runs under
Rosetta 2 on Apple Silicon (SWE-bench Docker images are x86_64), which
burns ~10 minutes per task. A full Verified sweep (500 tasks × 3
routes = 1500 runs × 10 min = ~250 hrs) is infeasible for our overnight
budget. We explicitly **do not claim** to measure the Verified
benchmark — we measure ten of its easiest tasks.

### 3.3 Category C part 1 — BigCodeBench-Hard (5 tasks)

| field | value |
|---|---|
| upstream | https://huggingface.co/datasets/bigcode/bigcodebench-hard |
| paper | Zhuo et al., *BigCodeBench* — [arXiv:2406.15877](https://arxiv.org/abs/2406.15877) |
| license | Apache-2.0 |
| adapter | `benchmark/bigcodebench_hard/adapter.py` |
| sample | 5 tasks from the 148-problem Hard slice, `seed=42` |
| scoring | functional (§5.1) |
| **contamination risk** | **MEDIUM-LOW** — released mid-2024, dataset structure is synthesized around library calls; specific prompts less likely to be memorized than HumanEval. |

**Why Hard:** state-of-the-art pass@1 on the full BigCodeBench sits
above 60% — quality deltas compress. The Hard subset keeps
headroom for routes to differ visibly.

### 3.4 Category C part 2 — custom-arch (5 tasks)

| field | value |
|---|---|
| upstream | none — hand-written in this repo |
| adapter | `benchmark/custom_arch/adapter.py` |
| tasks | multi-tenant auth design; zero-downtime migration plan; flaky-test code review; cache-invalidation tradeoff; production debug reasoning |
| scoring | bias-corrected pairwise LLM-judge (§5.3) |
| **contamination risk** | **ZERO** — unpublished, written for this project. |

**Why we wrote our own:** cat A/B are functional (pass/fail on tests);
they miss the coding work where "quality" is a judgement call —
architecture, debugging, trade-off discussion. Cat C part 2 probes
exactly that. **Bias risk** from hand-curation is acknowledged in §7.

## 4. Routing strategies

All three routes go through the same local proxy
(`router/server.mjs`), which records token counts per backend in
`X-Tokens-*` response headers. This keeps all three routes comparable
byte-for-byte on inputs.

| route | description | runner | router model arg |
|---|---|---|---|
| R1 | single POST to cloud model | `runners/r1_cloud_only.py` | `router/always-cloud` |
| R2 | single POST to local model | `runners/r2_local_only.py` | `router/always-local` |
| R3 | cloud planner → per-step heuristic executor → cloud synth | `runners/r3_hybrid_architect.py` → `runners/_architect_runner.mjs` → `router/agentic/architect-core.mjs::runArchitect` | per-call; planner forced `always-cloud`, executor `heuristic`, synth `heuristic` |

**Cloud model**: `gpt-5.5` (OpenAI, date-resolved to
`gpt-5.5-2026-04-23` at query time — see
`router/server.mjs::CLOUD_MODEL`). Fallback: `gpt-5` on 404/400.

**Local model**: `qwen3-coder:30b` via Ollama
(`router/server.mjs::LOCAL_MODEL`). MXFP8 quantization, running on
Apple Silicon MLX. Temperature=0 for determinism where the backend
respects it (Ollama does; OpenAI reasoning models ignore it).

**Router model (R3 only)**: `qwen3:0.6b` — a tiny classifier the
`heuristic` executor can consult for per-step routing. In practice
R3's planner emits `router_hint` values (`local`/`cloud`/`any`) which
short-circuit the classifier for most steps.

R3's token attribution is per-call: every call made by the architect
pipeline is tagged `planner/cloud`, `step_N/local|cloud`, or
`synth/cloud` in `routing.per_call_backends`, and tokens are summed
into `local_*` vs `cloud_*` buckets accordingly.

## 5. Scoring

### 5.1 Functional scorer (Categories A, C-BigCodeBench)

`scorers/functional_python.py` — extracts the first Python code block
from the model response, writes it to a tempfile plus the task's own
test file, runs pytest in an ephemeral Docker sandbox
(`scorers/Dockerfile.functional_python`). Composite score is
`tests_passed / tests_total` ∈ [0, 1]. A task with zero tests
returned counts as failed.

**Sandbox policy** (`lib/sandbox.py`):

| knob | value |
|---|---|
| image | `python:3.11-slim` + pip'd deps per benchmark |
| `network_mode` | `none` |
| `mem_limit` | `512m` |
| `pids_limit` | `128` |
| wall clock | 60 s default (per-task override allowed) |
| stdout/stderr | truncated to 16 KB (tail-preserving) |
| cleanup | container force-removed after each run |

### 5.2 SWE-bench scorer (Category B)

`scorers/swebench.py` — extracts a unified diff from the model
response, applies via the mini-swe-agent harness against the task's
pre-built Docker image, runs the project's own test suite. Binary
pass/fail; composite score in [0, 1] is 1.0 on `resolved=true`, 0.0
otherwise.

Known constraint: SWE-bench official images are x86_64. On M4 Max we
run them under Rosetta 2 / `qemu`, which is ~3× slower than native.
This does not affect correctness, but it bounds how many tasks we can
score in an overnight window.

### 5.3 LLM-judge scorer (Category C-custom-arch)

`scorers/llm_judge.py` — bias-corrected pairwise judge, adapted from
MT-Bench's rubric (we vendor the reference code at
`vendor/lm-eval-harness-judge/` but do not import it; we reimplement
so the rubric can be per-task).

| design decision | rationale |
|---|---|
| Judge model: `claude-opus-4-7` | Cross-vendor — our candidates are GPT-5.5 routes; using a Claude judge avoids the self-preference bias documented in Zheng et al., *Judging LLM-as-a-Judge*, [arXiv:2306.05685](https://arxiv.org/abs/2306.05685). |
| Two orderings per pair (A-vs-B + B-vs-A), averaged | Cancels position bias. |
| 5-dimension rubric per task | Per-axis scores averaged independently of the overall verdict, giving a stable signal when the overall vote flips. |
| Disagreement between orderings → tie | Margin reported at half the mean so flipped verdicts are visibly weak. |
| All judge reasoning persisted | Downstream readers can audit — see `results/judgments/` per run. |

Pairwise results are aggregated into a quality score in [0, 1] via
`judge_to_quality`: win-rate across the route's pairings.

## 6. Cost derivation

Tokens are recorded per call; dollars are derived. The same
`ResultRow` can be re-costed under a new pricing table without re-running
any benchmark.

| stage | location |
|---|---|
| Pricing table | `lib/pricing_tables.json`, `_meta.fetched_at = 2026-04-27T20:00:00Z`, source models.dev |
| Cost formula | `lib/pricing.py::compute_cost` — `(prompt - cached) * input + cached * cache_read + completion * output`, completion already includes reasoning tokens |
| Multi-scenario sweep | `lib/results.py` — 5 pricing scenarios (current, 25% cheaper, 50% cheaper, 2× more expensive, cache-enabled) so readers can see sensitivity |
| Local cost | **$0** for local-backend tokens (Ollama/MLX) |

The "local = $0" assumption is a simplification. It ignores:

- electricity (~1 kWh per hour of sustained GPU work, ~$0.15–0.30 at
  US retail);
- hardware amortization (an M4 Max 64 GB is ~$4000; over 3 years and
  40 hrs/week of coding that's ~$0.64/hour baseline);
- the opportunity cost of running a local model versus doing
  something else with the laptop.

A back-of-envelope marginal cost for local inference on our rig is
~$0.20–0.40/hour. On a task that takes 90 s that's ~$0.005–0.01 per
task — small compared to the cloud call it's replacing but **not
zero**. The headline numbers treat local as free; the "local
marginal cost" sensitivity scenario in the report adds back a
$0.30/hr charge.

## 7. Known biases and limitations

*This is the section reviewers should read first. Everything below is
a reason to discount our numbers.*

### 7.1 Single hardware tier

We ran every route on one machine: Apple M4 Max, 64 GB unified memory,
macOS 15. A 30B-parameter local model fits comfortably in RAM; it does
not fit on a 16 GB M2 Air or a 24 GB RTX 4090 (quantized). Results
will **not** transfer to smaller machines without re-running with a
smaller local model. Cloud-API latency from our network location also
affects R1 and R3 numbers; a user on a worse connection will see
different cloud wall-clock.

### 7.2 Single sample per (task, route)

N=1 per cell. Temperature=0 helps with local determinism (Ollama
respects it); it does not with GPT-5.5 reasoning models, which sample
their hidden-reasoning trajectories regardless of requested
temperature. We therefore **have no variance estimate for cloud
routes**. Treat point estimates as ±a bar of unknown width.

### 7.3 Contamination risk on Category A

HumanEval(+) predates every model we benchmark by 3–5 years. The
solutions are on GitHub. These models have almost certainly memorized
the problems. **Our Category A numbers should be read as "what it looks
like when the model already knows the answer"** — they are a floor
under which a route cannot credibly go, not a ceiling against which a
route's generalization can be measured.

### 7.4 LLM-judge biases

LLM-as-judge carries known biases (Zheng et al., 2023): length bias
(longer answers win), style bias (markdown/structure wins),
self-preference (a model prefers its own style). Our counter-measures:

- **bias-corrected pairwise**: A-vs-B and B-vs-A averaged, disagreement
  counted as tie;
- **cross-vendor judge**: `claude-opus-4-7` scoring `gpt-5.5` outputs
  removes the worst of the self-preference effect;
- **transparent rubric**: 5 named dimensions per task, rubric printed
  in the judge prompt;
- **reasoning persisted**: every verdict has a written rationale,
  auditable after the fact.

None of this eliminates bias — especially length bias. A longer
answer still tends to win. We report this and move on.

### 7.5 Small N per category

10 tasks per category is not enough for statistical significance. A
5-point quality delta on N=10 is two tasks. We report medians and
means, and we explicitly **do not** compute significance tests or
bootstrap CIs — the apparatus would imply precision we don't have.

### 7.6 Hand-curated tasks (custom-arch) bias

We wrote the 5 custom-arch tasks. We might — consciously or not — have
picked tasks that favour the routing strategy we were designing. We
mitigate by:

- publishing the full rubrics in `benchmark/custom_arch/tasks.jsonl`
  (no hidden scoring);
- using a cross-vendor judge (§5.3);
- flagging this limitation prominently (you're reading it).

If a skeptic wants to rewrite the rubric and re-judge, the data and
code support that.

### 7.7 Synth-budget exhaustion on reasoning models

During smoke tests (T4.1) we observed that `gpt-5.5`, when run in
reasoning mode on the R3 synthesis step, sometimes burns its entire
completion budget on **hidden** reasoning and returns an **empty
visible** message. The completion_tokens count is high; the body is
empty. Setting `max_tokens=6000` on synth calls usually avoids this,
but not always. Rows where this happened are surfaced as
`quality=0, error="empty_synth_visible_content"` rather than silently
scored 0. Count of affected runs is reported in `REPORT.md`.

### 7.8 SWE-bench harness runs under Rosetta

See §5.2. We measured 10 of ~500 Verified tasks. Do not read our
numbers as "Verified resolution rate"; they are a small easy-tier
sample.

### 7.9 Benchmark ≠ real development

The largest limitation, and the hardest to fix: our 30 tasks are
curated, self-contained, and presented as a single prompt. Real
coding work involves:

- a codebase context window (hundreds of files, not one);
- tool use (LSP, grep, repl, test-runner loops);
- iteration (the first response is rarely the final answer);
- debugging from error output;
- reading documentation the model has to fetch.

None of this is in our evaluation. A hybrid router that wins on 30
curated tasks might lose the moment it has to handle a 500 KB context
plus 4 tool calls. We flag this as the single largest threat to
external validity. Future work (§10) proposes developer-session
replays as a partial fix.

## 8. Reproducibility

| artefact | where |
|---|---|
| task samples (pinned) | `benchmark/*/tasks.jsonl`, `seed=42` baked into each adapter |
| code version | committed per phase on branch `wip-overnight-build` |
| env manifest | `results/<run-id>/env-manifest.json` — macOS version, Python, Node, Ollama model digest, git SHA |
| pricing table | `lib/pricing_tables.json` with dated `_meta.fetched_at` |
| raw rows | `results/<run-id>/rows.jsonl` — `ResultRow` per (task, route) |
| how to run | `REPRODUCING.md` (T6.2) |

Every `ResultRow` is self-describing — it carries the benchmark ID,
task ID, route, token breakdown, and quality score. Re-running the
pricing sweep a year from now requires only the rows and a new
pricing table.

## 9. Statistical caveats

- We report **medians** (resistant to outlier tasks) and **means**
  (for additivity). We do **not** report significance tests on N=10.
- **Bounded-ARQGC** (quality-per-dollar, bounded on the quality side
  so a 99.9%-quality route doesn't get infinite ARQGC) is a
  single-number summary. It is published **alongside** the per-task
  data, never instead of it. Formula in `lib/results.py::arqgc`.
- **Quality** is a composite on [0, 1]. Functional: pass-rate
  (binary per test, averaged within a task). Judge: pairwise win-rate
  aggregated to a row-level quality.
- We report a confidence-interval *band* on the main chart that
  reflects the worst-case 1-task swing (±10% on N=10) — this is a
  visual caveat, not a t-test.

## 10. What would strengthen this in a v2

1. **Multiple hardware tiers** — at minimum one mid-range laptop (M2
   Air 16 GB) and one mid-range GPU (RTX 4090 24 GB) with
   appropriately sized local models per tier.
2. **N ≥ 30 per category**, with repeated samples per (task, route)
   so variance is estimable.
3. **Developer-session replays** — batched trajectories from real
   coding sessions, not curated single-shot tasks.
4. **More benchmarks** — Aider Polyglot (multi-language, fresh),
   LiveCodeBench (rolling fresh problems to sidestep contamination).
5. **More routes** — R4 hybrid-minion (local-first with cloud
   escalation on low confidence) and R5 local-plus-cloud-review
   (local draft, cloud critique).
6. **Prompt-caching-enabled variant** of R1/R3 — caching the planner
   context across steps changes the cost math significantly.
7. **Fine-tuned local-model variants** — a Qwen3-Coder-30B
   fine-tuned on the developer's own repo would probably change the
   local-vs-cloud break-even meaningfully.
8. **Per-token energy accounting** — a real local-cost number
   derived from wall power measurement, not a flat $0 or $0.30/hr.

None of this is in scope for the current work. This document exists
so that future work can build on a clear record of what we did and
what we knew we weren't measuring.
