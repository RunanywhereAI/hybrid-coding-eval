Should I run my coding tasks local, hybrid, or cloud? A token-first answer. v3.3 sweep — 3,581 rows across 6 local models × 5 routes × 7 routing strategies × 8 task shapes.

📅 Published 2026-05-18 🏷️ tag v3.3 💻 M4 Max 64 GB ☁️ gpt-5.5 cloud ⚖️ claude-opus-4-7 judge github.com/RunanywhereAI/hybrid-coding-eval

TL;DR — three answers in 30 seconds

1. Can hybrid routing save cost? Not via multi-step orchestration (R3/R4/R5) — they cost 1.9× to 5× more than R1 cloud-only with equal or worse quality. Yes via per-task gating (try local first, fall back to cloud on failure): ~16–20% savings on a mixed workload.

2. Best local model? Qwen3-Coder:30B at $0.229/correct — beats devstral, qwen2.5-coder, gemma4, GLM-flash, AND both newer Qwen 3.6 variants (the surprise: newer ≠ better when specialization is dropped).

3. Best routing strategy? Cascade with default threshold 15 — replicates as Pareto winner across all 6 models. The much-anticipated llm-classifier strategy is structurally broken on SWE-bench: 5 classifier sizes from 0.6B to 4B all score 0/10. Don't deploy it.

The fundamental cost question

A real developer's mixed workload contains 8 task shapes: tiny completions, real GitHub patches, library calls, architecture prose, refactors, code reviews, scripts. The router's job is one decision per task: can local alone solve it? The v3.3 data lets us quantify the answer per shape.

The 5 most important findings

Finding 1
Qwen3-Coder:30B is the universal local-model winner
$0.229/correct on R3 heuristic — 14% cheaper than 2nd place, 22% cheaper than devstral baseline. The newer Qwen 3.6 family regressed: both 27B-mxfp8 and 35B-MoE underperform.
Finding 2
Cascade routing wins across all 6 models
~15% cheaper than heuristic at quality parity. Replicates universally. Default threshold 15 is empirically optimal (Phase 7 sub-sweep: t=20 is a brittleness cliff at -21% pass rate / -80% on SWE-bench).
Finding 3
LLM-classifier is structurally broken on SWE-bench
5 different classifier sizes (0.6B → 4B) tested in Phase 6. All score 0/10 on B-category. Scaling does NOT fix it. Code-specialized variant provides no advantage. Don't deploy.
Finding 4
GLM-4.7-Flash has a category-A collapse
0/10 on HumanEval+ on every route. But 5/5 on architecture prose (C-arch). Reasoning-optimized model can't generate one-line code solutions. Niche workload only.
Finding 5
Multi-step hybrid (R3/R4/R5) always loses to R1
R3 = 2.26× R1 cost. R4 = 1.91×. R5 = 5.13× (and collapses on prose). Architect-prefix + synth overhead dominates per-step local savings on small/medium tasks.

The deployable router design

Based on all v3.3 findings, the deployable router is not R3, R4, or R5. It's a per-task gating architecture:

Task arrives
    ↓
Stage 1: Cheap heuristic gate (0 ms)
    ├─ score < 25? → try R2 (local Qwen3-Coder:30B)
    │      ├─ success test passes? → return ($0)
    │      └─ fail? → fall through to R1
    └─ score ≥ 25? → direct R1 (cloud gpt-5.5)
